# PRD 02 · M1.5 成就与打磨

> 前置依赖：本 PRD 完全建立在 **01-m1-core-farm-loop.md**（M1 农场核心循环）交付之上——地图/移动/6 作物/时间与夜结算/出货箱与商店/金币与 Lv1~5/12 格背包（仅点击移动）/成就**计数器全量埋点**（无成就 UI）/存档与导入导出，均为 M1-core 既有事实。00 号工程基建 PRD（monorepo 脚手架/CI/lint）为本 PRD 的公共前置（亦经由 01 传递满足）。后续 04~06 号 PRD（M3 建造 / M4 AI 关卡 / M5 发布）依赖本 PRD 交付的成就引擎数据表与重型测试资产；03（M2 HUD）与本 PRD 无依赖、可并行。

## Problem Statement

M1-core 交付后，核心循环（种→收→卖→升级）已经成立，但作为「一个真正好玩的农场游戏」仍有三处明显欠账：

1. **劳作缺少确认感**：玩家的第一次锄地、第一次收获、第一百次收获都悄无声息。计数器在 M1-core 已经默默累计，但没有任何成就把这些时刻标记出来——休闲玩家（只浇水、顺手收的「候车室」玩家）尤其缺少「我在前进」的低压反馈。
2. **中后期操作手感粗糙**：玩家在 D10 前后买到铜工具时，工具行为和木档没有任何区别可感（范围交互未实装），「350g 花得值不值」无法被第一时间确认；背包只有点击移动，没有拖拽、没有误删反悔、没有一键整理，12 格背包的日常整理很费键。
3. **质量没有被「重型验收」锁住**：确定性（同 seed 可复现）、天气分布（万夜统计）、背包不变量（fuzz 物品守恒）以及红线 1 的真人 playtest（未读文档的新玩家 10 分钟内完成首次种→收→卖并升 Lv2）都还只是设计文档上的承诺，没有变成可以反复执行的测试资产。在这些回归网建立之前宣布 M1 发布，等于把质量债带进 M2~M5。

## Solution

按 game-design.md §0.4 的 M1.5 行，原样交付四块内容（不多不少）：

1. **成就系统（#1~#14 可解锁 + 简版成就页）**：22 条成就全部以数据表定义入引擎（id 与条件以 §5.6 为唯一事实源），本里程碑只有 #1~#14 可解锁；解锁提示走右下角 2.5s 非模态 toast，Esc 菜单新增「成就」tab 展示已解锁项与未完成项的实时计数。成就奖励（XP+金币）与经济推演**严格解耦**：脚本 R 与红线 1 的验收口径不含任何成就奖励（§4.6、附录 B-3）。
2. **铜/金档工具范围交互与升级视觉反馈**：锄头铜/金档长按 = 半透明范围预览 + 松开批量（防误锄）；水壶全档位 280ms 连发、每拍作用当前档位范围；铜/金档选中时常显面朝方向范围框；升级后第一次挥动即有强烈视觉差异（更宽水弧、挥动残影、湿土扩散）。
3. **背包打磨三件套**：拖拽（光标手持）状态机、垃圾桶撤销槽（单步反悔、不入存档）、R 键整理（只动储备格不动快捷栏）。
4. **重型验收与红线校准**：确定性回放 ×100、万夜（10,000 夜）天气模拟、拖拽 fuzz 1,000 次进入测试套件；执行红线 1 真人 playtest（D3 晨 Lv2 / 10 真实分钟），校准只允许动引导与文案，数值改动必须先回 GDD 修订。

M1.5 与 M1-core 合并构成「M1 发布」；本 PRD 全部验收绿 + §0.5 红线 1~3 成立 = M1 出厂。

## User Stories

### A. 成就系统——解锁与反馈（玩家 = 等待 AI 的程序员）

1. 作为玩家，我想要第一次锄地、第一次播种、第一次收获、第一次（夜结算）售出时分别解锁「破土」「第一粒种子」「第一次收获」「第一桶金」，以便我最初的每一个动作都得到一次明确的确认。
2. 作为玩家，我想要成就解锁时只在屏幕右下角出现一条 2.5 秒的非模态 toast（不弹窗、不暂停时间、不抢键盘焦点、不移动镜头），以便我在等 AI 的低压时段绝不被游戏自己打断（设计原则 2「感知不干预」在游戏内的同构）。
3. 作为玩家，我想要第一次经历雨天就解锁「雨天的馈赠」，以便「下雨自动浇水、今天可以偷懒」这条福利规则被一次性地明确教给我。
4. 作为玩家，我想要第一次过夜解锁「过夜」、累计金币达 1,000/10,000 解锁「小有积蓄」「千金」、百次收获与 200 次浇水各有成就，以便日常劳作自然累积出长期里程碑，而我不需要为成就改变任何玩法。
5. 作为玩家，我想要升级第一件工具解锁「装备升级」、两件工具都到金档解锁「黄金装备」，以便 6,000g 的工具长线投资有非金钱的仪式性回报（工具升级本身 0 XP，防「花钱=刷级」，§5.2）。
6. 作为玩家，我想要 6 种起步作物各售出至少 1 次时解锁「初识六谷」、再生作物连收 4 茬解锁「再生行家」，以便「每种都试试」「长期照料豆藤」这类探索行为被奖励。
7. 作为玩家，我想要带金币的成就（#3/#4/#8/#13，合计 340g）解锁时金币**即时入账**（不走出货箱过夜），以便一次性奖励的爽点不被延迟稀释。
8. 作为玩家，我想要成就的 XP 奖励走与播种/收获完全相同的 XP 管线（头顶飘字、XP 细条生长、升级横幅、M1 的 Lv5 等级帽一视同仁），以便所有进度反馈语言一致。
9. 作为玩家，我想要所有成就都是累计型、永不过期、永不可失败、与真实时间完全无关（无连续登录/限时/赛季，§5.6 原则①②），以便我随时离开几天再回来，不存在任何「错过」。
10. 作为玩家，我想要同屏提示遵守「最多 1 条升级横幅 + 1 条成就 toast、FIFO 队列上限 8、溢出合并为『还有 N 项新进展，详见日结算』、模态界面打开时暂停弹出」（§5.8），以便集中触发（例如导入老档补结算）时不被刷屏。
11. 作为玩家，我想要当日新解锁的成就汇总出现在日结算屏的进度区块里，以便睡前回顾「今天有什么新进展」。

### B. 成就系统——成就页与持久化（玩家）

12. 作为玩家，我想要在 Esc 菜单里新增的「成就」tab 看到简版成就页：已解锁项（名称+达成条件+奖励）与未完成项的实时计数（如「百次收获 37/100」），以便随时知道每项还差多少——但页面永不显示倒计时或每日任务。
13. 作为玩家，我想要 M3/M4 才能推进的成就（#15~#22：建筑、职业、quest 等）**不出现**在 M1 成就页里，以便页面上没有任何当前版本「永远做不完的待办」（与 §5.3「M1 中 Lv6+ 条目一律折叠」同一纪律）。
14. 作为玩家，我想要成就与计数器随存档自动保存、随 JSON 导出/导入完整往返，以便换浏览器或清站点数据后我的成就履历不丢。
15. 作为玩家，我想要导入的存档里出现本版本不认识的成就 id 时被原样保留而不是删除（§5.8 边界），以便存档在版本间往返不损失记录。
16. 作为玩家，我想要成就解锁严格幂等——同一成就只解锁一次、只发一次奖励，读档/导入/重复触发都不会重复发奖，以便经济与 XP 不被存档操作污染。

### C. 铜/金档工具范围交互（玩家）

17. 作为玩家，我想要买到铜锄头后，长按（≥400ms）出现「直线 3 格」的半透明范围预览、**松开才批量执行**，非法格（不可开垦/已开垦/达开垦帽）自动跳过只作用合法子集，以便我在「开垦帽计数单调不可逆」的世界里永远不会误锄（§3.5、A-16）。
18. 作为玩家，我想要金锄头同样的预览-批量交互、范围为 3×3（面朝格为中心 9 格），以便后期整田一次成型。
19. 作为玩家，我想要锄头**轻按永远只作用 1 格**，不随档位改变，以便升级不破坏我已经形成的肌肉记忆。
20. 作为玩家，我想要铜/金水壶保持全档位 280ms 长按连发、每拍作用当前档位范围（直线 3 / 3×3），已湿格与非耕地格 no-op 跳过且不打断节拍，以便连浇一整片田又快又顺、不打嗝（§3.5、§1.10-8）。
21. 作为玩家，我想要手持铜/金档工具时面朝方向常显一个半透明范围框（§6.4），以便挥下去之前就确切知道会作用哪些格子。
22. 作为玩家，我想要工具升级后的第一次使用就能看到明显的视觉差异——更宽的水弧、挥动残影、湿土瞬间扩散动画（§3.5），以便 350g/2,650g 的购买决策被第一时间确认「花得值」。
23. 作为玩家，我想要 22:00 入睡边界打断锄头长按蓄力时，范围预览直接取消、不执行、零惩罚（§3.9-4），以便边界时刻不产生意外开垦。
24. 作为玩家，我想要范围批量开垦后右上角「耕地 17/18」计数器一次性正确更新、达帽时闪烁一次（§1.4/§5.8），以便范围操作下开垦帽依然清晰可感。
25. 作为玩家，我想要键盘（E 长按）与鼠标（左键长按）的范围交互产生完全相同的 sim 命令序列（§1.10 验收），以便纯键盘党与鼠标党手感等价。

### D. 背包打磨：拖拽 / 撤销槽 / 整理（玩家）

26. 作为玩家，我想要在背包界面用左键拿起整堆、右键拿起 ⌈n/2⌉ 半堆，手持时左键放整堆/同 id 合并至 99 溢出留手/异 id 交换、右键放 1 个，以便用熟悉的「光标手持」方式快速摆放物品（§6.7 状态机）。
27. 作为玩家，我想要 Shift+左键快速移动（在商店卖出 tab 等于整堆寄售），以便大宗操作一步到位。
28. 作为玩家，我想要手持物品时按 Esc 或关闭界面，物品自动归还原格（原格被占则进第一空格）——任何路径下物品绝不消失（§6.7 / §6.9），以便我从不担心手滑丢东西。
29. 作为玩家，我想要丢进垃圾桶的物品出现在「撤销槽」里、点击即取回最近一次销毁的堆（再次丢弃/关背包/日结算时清空，不入存档，§6.3），以便误删可以单步反悔，而游戏因此不需要任何丢弃确认弹窗。
30. 作为玩家，我想要 R 键一键整理：合并同类堆、按 `tool < seed < crop < material < quest`、同类按 id 字典序排序，且**只整理储备格、绝不动前 9 格快捷栏**（§6.2），以便整理背包不破坏 1~9 键的肌肉记忆。

### E. 验收红线校准与发布质量（玩家 / 开源使用者）

31. 作为玩家（从未读过文档、没玩过农场游戏的新玩家），我想要在无任何口头提示下 10 真实分钟内自然完成首次「种→收→卖」并升到农场 Lv2（§0.5 红线 1、§1.9 动线），以便游戏第一性（可玩性独立成立）被真人验证而非只有 bot 推演。
32. 作为玩家，我想要红线 1 的达成**不依赖任何成就奖励**（D3 晨 Lv2 由播种 50 XP + 收获 60 XP 独立构成，§5.4），以便成就系统怎么调都不会破坏新手节奏。
33. 作为玩家，我想要任意 2 分钟的游玩至少包含「浇完水 + 看到一个进度条前进」（红线 2，由入箱金额/生长进度/成就计数共同承担，A-3），以便短会话也有完整的小闭环。
34. 作为开源使用者，我想要 M1 发布版（M1-core + M1.5）在同一 seed 与同一操作脚本下逐字节可复现（§2.9/§3.9），以便我提 bug 时能附上可复现的种子与序列，维护者能稳定重放。
35. 作为开源使用者，我想要 M1.5 新增的全部视觉资产（范围框、水弧、残影、成就图标等）继续满足许可白名单（CC0-1.0 / OFL-1.1）并可在 manifest 中逐文件追溯（§11.1、红线 5），以便仓库可以放心 clone 与再分发。
36. 作为开源使用者，我想要成就与全部进度数据只存在我的本机存档里（无遥测、无上报），以便隐私承诺（设计原则 3）在游戏侧同样成立。

### F. 后续开发者（含 AI agent）

37. 作为后续开发者，我想要 22 条成就全部以**数据表 + 纯函数谓词**形式定义在 sim 进度模块（id/条件/奖励逐字对齐 §5.6 表），M3/M4 只需让对应计数器开始跳动、把里程碑标记翻开，零引擎改动即可点亮 #15~#22，以便成就系统一次成型、后续只做数据接线。
38. 作为后续开发者，我想要「依赖等级/XP 的成就（#21/#22）XP 必须为 0」「#1~#14 XP 合计 = 305」「成就金币合计 = 340g」这三条不变量有单测看护（§5.6），以便未来任何调表都不会引入 XP 反馈环或冲垮经济带宽。
39. 作为后续开发者，我想要脚本 R 推演「不含成就奖励」的口径在 sim 测试中固化为显式运行模式（§4.6、B-3），以便成就数值与经济验收带宽永久解耦、互不回归。
40. 作为后续开发者，我想要对照 §5.6 计数器清单（tillCount/plantCount/harvestCount/waterCount/sellCount/goldEarned/soldCrops:&lt;cropId&gt;/sleepCount/rainDaysSeen/toolUpgrades/regrowChainMax 等）逐项审计 M1-core 的埋点完备性（每个 M1 可触发计数器至少一条触发测试），以便成就上线时不存在「永远无法解锁」的死成就。
41. 作为后续开发者，我想要确定性回放 ×100、万夜天气模拟、拖拽 fuzz 1,000 次成为可在 CI 反复执行的常规测试资产（全部 headless、零 Phaser），以便 M2~M5 的任何重构都有回归网兜底。
42. 作为后续开发者，我想要拖拽状态机实现为纯 store 逻辑（渲染层只画光标贴图），以便 fuzz 测试无需浏览器即可跑一千次随机序列并断言物品守恒。
43. 作为后续开发者，我想要 quest 累计 5/15/30 道具奖励（§9.8）的处理决策被本 PRD 明确记录——schema 不加 item 字段、M4 按无道具实现、道具与图鉴形态待 M3 定稿，以便 M3/M4 的 PRD 与实现者无需重新考古。
44. 作为后续开发者，我想要 playtest 校准的「可调面/不可调面」边界成文（只许动引导与文案；数值与状态机改动必须先修订 GDD），以便事实源永不漂移（GDD 尾注纪律）。
45. 作为后续开发者，我想要 M1.5 对存档 schema **零变更**（成就/计数器字段在 v1 已就位，§10.2），以便 schemaVersion 升版与迁移链整体留给 M3 引入 v2 时一次建立（§10.6）。

## Implementation Decisions

以下决策均以设计定稿为唯一事实源；本节只内联 load-bearing 的状态机与数据形状（均注明出处），数值表一律引用章节号。

1. **范围铁律**：本 PRD 的 In/Out 逐字遵循 game-design.md §0.4 M1.5 行——成就 #1~#14 + 简版成就页；铜/金档工具范围交互与升级视觉反馈；拖拽状态机/撤销槽/R 整理；确定性回放 ×100、万夜天气模拟、拖拽 fuzz 1,000 次等重型验收。M1-core 的 Out 清单与「移出 M1」清单原样继承，不得回流。

2. **模块划分与接口**：
   - **sim 进度模块（纯函数，零 Phaser、零墙钟）**：持有 22 条成就数据表与判定/发奖逻辑；对外仅 `bumpCounter`、`grantXp`、`checkAchievements` 与既有事件（`achievement:unlocked` / `xpGained` / `levelUp`），数据结构沿用 §5.5 的 `ProgressionState`（xp/counters/achievements/xpHistory，achievements append-only）。
   - **输入转译层**：铜/金档的范围展开（直线 3 / 3×3）在输入层完成，sim 只接收既有 `FarmAction { kind:'till'|'water'; tiles: TilePos[] }`（§3.8 既定形状，本里程碑零 sim 接口变更）。
   - **UI 壳层**：成就 toast 与提示队列、Esc 菜单「成就」tab、范围预览/范围框渲染、升级视觉反馈（水弧/残影/湿土扩散）、拖拽光标层。全部为薄渲染，不持有真值。
   - **背包 store**：拖拽状态机与撤销槽是纯 store 状态（UI 态，不入存档）；`InventoryApi`（§6.2：add/canAdd/removeAt/move/select/sort）保持不变，`sort()` 在本里程碑实装。

3. **成就数据表形状**（工程化自设计定稿 §5.5/§5.6）：

   ```ts
   // id 集合 = §5.6 表 22 个 id 逐字采用（first_till … mastery）
   interface AchievementDef {
     id: AchievementId;
     milestone: 'M1.5' | 'M3' | 'M4';            // §5.6 表「实装」列
     predicate: (view: ProgressView) => boolean;  // 多数 = 单计数器阈值；
                                                  // #12 gilded = 双工具档位谓词；
                                                  // #13 six_crops = 6 个 soldCrops:<cropId> 联合谓词
     reward: { xp: number; gold: number };        // 数值以 §5.6 表为准，不在此复制
   }
   ```

   判定触发点 = 每次 `bumpCounter` 之后与工具升级事件之后；`ProgressView` 是 sim 状态的只读视图（计数器 + 工具档位）。解锁即追加进 `achievements` 数组（append-only ⇒ 幂等）；`milestone !== 'M1.5'` 的条目在 M1 构建中谓词永不满足（触发面不存在），仅作为数据就位。

4. **奖励发放契约**：成就 XP 经统一 `grantXp` 管线发放（受 §5.1/§5.3 的 `effectiveLevel = min(levelForXp(xp), 5)` M1 等级帽约束，XP 本身不封帽）；成就金币**即时入钱包**、不走出货箱（§4.7 将其列为一次性 faucet，总额 340g，「M1.5 起；不入红线 1 与脚本 R 推演」）。每条成就的奖励终身发放一次，幂等性由 achievements append-only 保证。

5. **成就与经济推演解耦（本 PRD 的中心裁决，来自 §4.6 / §5.4 / 附录 B-3）**：脚本 R 与红线 1 的全部带宽断言在「成就奖励关闭」的推演模式下执行（sim 推演入口提供该显式模式）；D3 晨 Lv2 由播种 +50 / 收获 +60 XP 独立构成，**不依赖任何成就**。注意：附录 A-1 文末「红线 1 推演……依赖成就 #3/#4 的 +40g」是合稿期历史记录，已被其后的 B-3 裁决与 §4.6/§5.4 现行口径取代——实现一律以后者为准。

6. **成就页与提示呈现**（§5.8 为权威）：简版成就页 = Esc 菜单新增 tab，仅列出 #1~#14（已解锁项 + 未完成项实时计数如「37/100」），#15~#22 不在 M1 UI 出现；永不显示倒计时。成就 toast 固定右下角 2.5s；提示队列规则：同屏最多 1 横幅 + 1 toast、FIFO 上限 8、溢出合并文案、模态打开时暂停弹出、多级连升每级独立横幅间隔 0.5s。成就 toast **不新增音频资产**（M1 音频收敛为 §11.5 的 8 个 SFX；可复用其中之一或静默，专属音效留 M3 音频管线）。

7. **5/15/30 道具奖励的处理（任务点名项，裁决来自 §9.8）**：quest 累计完成 5/15/30 个的道具奖励（种子礼包/沉思的稻草人/村民的信）已定为「道具与图鉴形态待 M3 图鉴/装饰子系统定稿后确认，在此之前 M4 按无道具实现」。本 PRD 据此固化三点：① M1.5 的成就/奖励数据形状**不加任何 item 字段**——§10.6 规定任何字段增删必升 schemaVersion，而 v2 与迁移链整体留给 M3；② 同理，成就 #21（纪念雕像）/#22（金色牌匾）的实物奖励随其成就一并归 M3；③ 本条仅为文档锁定口径，M1.5 无对应实现工作量。

8. **工具交互语义**（附录 A-16 统一时序为唯一事实源：动作锁 250ms / 缓冲 150ms / 长按阈值 400ms / 连发 280ms）：
   - 语义分档：**木档长按 = 连发；铜/金锄头长按 = 范围预览 + 松开批量；水壶全档位长按 = 280ms 连发、每拍作用当前档位范围**；
   - 轻按：锄头永远 1 格；水壶 = 当前档位范围一次（已湿/非耕地 no-op 跳过）；
   - 范围定义：「直线 3 格」= 面朝格 + 同向延伸 2 格；「3×3」= 面朝格为中心 9 格（§3.5）；
   - 边界：范围部分非法只作用合法子集（§3.9-3）；22:00 打断蓄力 → 预览取消、不生效、无惩罚（仅锄头有蓄力态，§3.9-4）；长按扫过无效格跳过不重置节拍（§1.10-8）；
   - 铜/金档选中时面朝方向常显半透明范围框（§6.4）；升级视觉反馈（更宽水弧、挥动残影、湿土瞬间扩散）随本里程碑落地（§3.5）。
   - 同格动作优先级表（§3.5）不变；工具价格/解锁等级引用 §3.5 与 §5.3，不复制。

9. **拖拽（光标手持）状态机**（来自设计定稿 §6.7，逐字采用）：

   ```
   EMPTY:  左键@有物格→拿整堆  右键@有物格→拿 ⌈n/2⌉
   HOLD(k): 左键@空格→放整堆 ｜ 左键@同id→合并至99溢出留手 ｜ 左键@异id→交换
            右键@空格/同id→放 1 个 ｜ Shift+左键→快速移动（卖出tab=卖整堆）
            点垃圾桶→discardable? 销毁→撤销槽 : 拒绝 ｜ Esc/关界面→归还原格→EMPTY
   ```

   归还规则补充（§6.7/§6.9）：原格被占则进第一空格，任何路径不消失物品；M1-core 的点击移动路径继续可用（拖拽是叠加而非替换）。

   > 裁决占位（M1.5 评审修复批次）：本条的右键拆堆语义与「InventoryApi 零变更」红线存在表达通道冲突——已以**新增** SimCommand `splitItem{from,to,count}` 落地（InventoryApi 六方法与存档 schema 均未动，红线维持）；通道形态待 owner 追认，见 docs/backlog/m1-review-backlog.md B-11。

10. **撤销槽与整理**：撤销槽显示最近一次被销毁的堆、点击取回；再次丢弃/关背包/日结算时清空；**不入存档**（§6.3）。R 整理 = 合并 + `tool < seed < crop < material < quest` + 同类 id 字典序，只动储备格（索引 9+）、不动快捷栏（§6.2）。`discardable:false`（工具等）维持双重拦截 + 摇头动画。

11. **存档**：M1.5 对 SaveDoc v1 **零 schema 变更**——`progress.counters` 与 `progress.achievements` 在 v1 已就位（§10.2，M1-core 埋点随档持久化）。导入含未知成就 id 的存档：保留不丢（§5.8 边界，向前兼容）。导入导出与校验口径沿用 §10.6 的 M1 简版。

12. **playtest 校准纪律（红线 1：D3 晨 Lv2）**：依 §0.5/§1.9 执行真人 playtest——未读任何文档、无农场游戏经验的新玩家，无口头提示，10 真实分钟内完成首次种→收→卖（出售完成时点 = D3 夜结算，≈9 分钟）并升 Lv2。失败时的**可调面**：新手引导三件套的文案与呈现（门廊信件、告示牌阶段提示、指路牌）、结算屏「明日之诺」条目、NEW 角标、toast 文案；**不可调面**：作物/价格/XP 数值表、状态机、时序常量——这些改动必须先修订 game-design.md 对应章节并通过其校验单测（GDD 尾注纪律），再回到实现。

13. **关键数值出处（不复制全表）**：成就条件与奖励 → §5.6 表；XP 阈值与等级帽 → §5.1；XP 来源 → §5.2；计数器清单 → §5.6；工具档位/价格/范围 → §3.5；统一时序 → 附录 A-16；脚本 R 与验收带宽 → §4.6；faucet/sink 台账 → §4.7；天气参数（春雨 20%、第 1 天强制晴、连雨 ≤2）→ §2.1/§3.4；提示队列与成就页 → §5.8；节奏带（勤奋 bot Lv2 ≤ D3.5、Lv5 ∈ D[22,27]；休闲 bot 28 天 ≥ Lv4）→ §5.4/§5.8 验收要点。

## Testing Decisions

总原则（tech-stack §1 裁决）：好测试只断言外部行为（状态输入 → 状态/事件输出），不窥探实现细节；渲染层不写单测。本里程碑用到的接缝与理由：

- **接缝 a：sim 纯函数层 headless 测试（最高接缝）**——成就引擎、XP 管线、范围动作、脚本 R 推演、天气模拟全部跑在 Node 里，零 Phaser、零真实时钟。理由：M1.5 的交付物有一半本身就是「重型验收」，只有纯函数层能以秒级成本跑万夜模拟与 ×100 回放。
- **接缝 a 的延伸：背包/拖拽纯 store headless 测试**——拖拽状态机实现为纯 store（实现决策 2），fuzz 1,000 次随机操作无需浏览器。理由：物品守恒是不变量断言，放进渲染层既慢又脆。
- **接缝 b：shared 包 zod schema 编解码测试**——含成就/计数器的存档 roundtrip 与宽容加载。本里程碑零 schema 变更，故只做回归级覆盖。
- 接缝 c/d/e（daemon 相关）本里程碑不适用（无 daemon 代码面）；接缝 f（Playwright 冒烟）按 tech-stack 裁决整体推迟 M5，本里程碑的视觉项（水弧/残影/范围框/toast 视觉）走人工验收清单。

测试资产清单（全部进 CI；随机化测试一律 seeded，失败时输出最小复现 seed/序列）：

1. **成就引擎**：阈值/谓词解锁正确性（含 #12 双工具谓词、#13 六作物联合谓词、#14 regrowChainMax）；解锁幂等（重复 bump/读档重放只发一次奖励）；achievements append-only；解锁与计数顺序无关。
2. **不变量看护**（§5.6）：#21/#22 reward.xp === 0；#1~#14 XP 合计 === 305；成就金币合计 === 340；锄/浇各 10,000 次 XP === 0（§5.5 验收）；XP 封顶 15,000 后 `xpGained(amount 0)` 仍发事件。
3. **解耦断言**：成就奖励关闭模式下，脚本 R 28 天推演逐项落 §4.6 验收带宽（D3=180 / D7≈324 / D14≈1,050 / D28 带宽），且与 01 PRD 的 M1-core 基线结果一致（成就引擎上线对推演零扰动）；成就开启模式下勤奋/休闲 bot 落 §5.4/§5.8 节奏带（Lv2 ≤ D3.5、Lv5 ∈ D[22,27]、休闲 28 天 ≥ Lv4）。
4. **确定性回放 ×100**：同 seed 同操作脚本快进 28 天，全程日志哈希一致，重复 100 次（§3.9 验收）；暂停期序列化状态逐字节相同（§2.9 回归）。
5. **万夜天气模拟**：10,000 夜统计——雨频 20%±3%、无 3 连雨、第 1 天恒晴（§2.9 验收口径）。
6. **拖拽 fuzz ×1,000**：随机操作序列（拿/放/合并/交换/半堆/快速移动/丢弃/撤销/关界面）后断言：物品总数守恒、无负数堆、无 >99 堆、撤销取回守恒、关界面后手持必空（§6.9 验收）。
7. **范围工具**：铜壶单次 ≤3 格直线、金壶 ≤9 格（§3.9 验收）；部分非法只作用合法子集；蓄力被 22:00 打断取消；E 与鼠标产生相同 sim 命令序列（§1.10 验收）；长按扫过无效格不重置 280ms 节拍；范围批量后开垦帽计数正确、达帽后续格被拒。
8. **整理与撤销**：sort() 只重排索引 9+、快捷栏 0~8 原位；排序符合类别与 id 字典序；撤销槽在再次丢弃/关背包/日结算三个时点清空且不入存档（存档前后状态对比）。
9. **存档往返**：成就/计数器导出→导入 derive 一致；未知成就 id 保留；坏档宽容加载不崩溃（回归）。
10. **埋点完备性审计**：§5.6 计数器清单中 M1 可触发项逐一有触发测试（种/收/浇/卖/睡/雨/升级工具/再生连茬），任何成就不可能成为「死成就」。
11. **人工验收清单（不写自动化）**：升级视觉反馈三件（水弧/残影/湿土扩散）肉眼可辨；范围框跟随朝向；成就 toast 位置/时长/不抢焦点；成就页全键盘可操作；提示队列溢出合并文案。
12. **playtest 协议**：≥2 名未读文档、无农场游戏经验的测试者；无口头提示；记录 10 分钟红线达成与全部卡点；如需调整仅按实现决策 12 的可调面修改文案/引导后复测；结论与改动记录归档（人工记录，无遥测）。

## Out of Scope

依据 game-design.md §0.4 里程碑拆分与附录裁决，以下明确不做：

- **M1-core 的 Out 清单原样继承**：体力、品质、动物、建筑、NPC、季节轮替、钓鱼/挖矿/战斗、会话 HUD（→ 各归 M2/M3/M4/M5 或永久 Out）。
- **「移出 M1」清单不回流**（§0.4）：AudioDirector / BGM 三首 / 连击音阶 / loudnorm 双格式音频管线 → M3；Web Locks 多标签互斥、backup/corrupt 恢复链、存储降级链、30s 脏轮存、导入预览屏 → M5；assets:import 自动化管线与 sha256 重现校验 → M5。
- **成就 #15~#22 的解锁路径与 UI 露出** → M3（#15~#18、#21、#22）/ M4（#19、#20）；本里程碑仅数据表就位。
- **图鉴 UI** → M3（`collectionLog` 数据 M1-core 已记录，§4.8）。
- **quest 累计 5/15/30 道具奖励的道具实现** → 道具与图鉴形态待 M3 定稿，M4 按无道具实现（§9.8 裁决）；**#21 纪念雕像 / #22 金色牌匾实物** → M3。
- **SaveDoc schemaVersion v2、迁移链与 CI 守护测试** → M3 引入 v2 时一次建立（§10.6）。
- **新音频资产 / 成就专属音效 / 镰刀升级档位** → M3。
- **Playwright E2E 冒烟** → M5（tech-stack 裁决）。
- **改键、手柄/触屏、英文文案、多存档槽** → M5 或附录 B-12 待裁决项。
- 任何 GDD 未载的成就周边功能（成就分享图、稀有度、隐藏成就、平台成就对接）一律不做。

## Further Notes

- **素材许可红线**：M1.5 新增的全部视觉元素（范围预览框、水弧/残影/湿土扩散帧、成就图标、成就页 UI 件）必须为程序化绘制、Kenney CC0 或自绘（以 CC0 献出），逐文件进 manifest 追溯（§11.1、红线 5）；绝不引用星露谷本体素材。
- **隐私红线**：playtest 为线下人工观察记录，不引入任何遥测/埋点上报；成就与计数器数据仅存本机 IndexedDB 存档与用户自己导出的 JSON。
- **文档一致性风险**：附录 A-1 的历史表述（红线 1 依赖成就 #3/#4 金币）与 B-3/§4.6/§5.4 的现行解耦口径表面冲突——实现以现行口径为准；建议 GDD 下次修订在 A-1 处加注「已被 B-3 取代」，避免后续 agent 按旧文实现。
- **节奏风险**：成就 #1~#14 共 305 XP（刻意偏高，拉休闲路径入带）可能让勤奋玩家略早达 Lv5；§5.8 验收带 Lv5 ∈ D[22,27] 已覆盖该效应，出带时按校准纪律回 GDD 修订，不得在实现侧私调。
- **回归风险**：拖拽状态机与 M1-core 点击移动并存，点击路径的既有测试必须保留并全绿；范围工具上线后木档单格路径同样保留回归。
- **CI 成本**：万夜模拟与 fuzz 均为纯函数级（预期秒级）；确定性回放 ×100 若超时可「抽样进 PR CI + 全量进 nightly」，属实现自由度，不改验收口径。
- **发布判定**：M1.5 与 M1-core 合并为「M1 发布」；出厂条件 = 本 PRD 全部测试资产绿 + §0.5 红线 1~3 成立（红线 4/5 分别随 M2/持续生效）。
