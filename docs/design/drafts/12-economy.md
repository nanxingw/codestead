## 0. 设计目标与三支柱对照

| 支柱 | 经济系统的承诺 |
|---|---|
| 游戏第一 | 经济是核心循环的「计分板 + 油门」：每天有挣钱的小决策（种什么、攒不攒工具钱），每级有新的花钱去处；定价全部由公式推导并被 sim 测试守护，可独立于监控语境成立 |
| 零焦虑承诺 | 无税收、无维护费、无贷款、无限时折扣、无每日特惠、无动态价格、无库存轮换——**商店里没有任何会错过的东西**；误买可全额退货、误投出售箱可取回、出售箱内容物永不变质 |
| 感知不干预 | 经济与会话 HUD 零绑定（宪法 §3）；M4 任务奖励是「捎带的谢礼」（≤日收入 ~10%，见 18-ai-quests §8），永远不值得为刷任务停止种地 |

## 1. 货币与钱包（wallet）

### 1.1 规则

| 项 | 规格 |
|---|---|
| 货币 | 单一货币**金币（gold，符号 g）**，M1~M5 不引入第二货币（材料是物品不是货币） |
| 取值 | 非负整数；新档 **100g**（宪法 §4.4）；上限 `GOLD_CAP = 9,999,999`（clamp，不溢出） |
| 负债 | 不存在。任何支出前置校验，余额不足 = 操作不可用（按钮置灰 + 缺口提示），不弹错误窗 |
| 舍入 | 所有乘数计价**对单件价格做一次 floor**（见 §5.5），先乘后舍，全程整数 |
| 显示 | 千分位分隔（`1,234g`）；金币显示位于右上角面板（归 16-inventory-ui §6.2，左上角永远留给会话 HUD） |

### 1.2 sim 层接口（`packages/game/src/sim/economy/wallet.ts`，零 Phaser 依赖）

```ts
export type GoldSource = 'shipping' | 'refund' | 'achievement' | 'quest' | 'salvage';
export type GoldSink = 'seed' | 'tool_upgrade' | 'backpack' | 'material' | 'building' | 'decoration';

export interface GoldChange {
  day: number;                    // 发生的游戏日
  kind: 'credit' | 'debit';
  amount: number;                 // 正整数
  source?: GoldSource;
  sink?: GoldSink;
  itemId?: string;
  count?: number;
}

export const GOLD_CAP = 9_999_999;

// 纯函数；溢出 clamp；余额不足返回 Err，由 UI 转译为置灰/提示
export function credit(gold: number, amount: number): number;
export function debit(gold: number, amount: number): number | 'INSUFFICIENT_GOLD';
```

每次变动发出 `economy:goldChanged { gold, change }` 事件：渲染层据此做 300ms 数字滚动 + 金币音（16-inventory-ui §6.2）；日结算屏据当日 `GoldChange[]` 聚合「今日收支」。事件日志仅保留**当日**（日结算后清空，不入存档）。

## 2. 出售设计：过夜结算（核心决策）

### 2.1 决策记录：出售箱过夜结算 vs 即时收购

**裁决：作物出售一律过夜结算；种子退货即时。** 比较过程：

| 维度 | A. 出售箱过夜结算（采用） | B. 即时入账（否决） |
|---|---|---|
| 与玩家画像的契合 | 宪法 §2：单次游玩「必跨至少一次过夜回报」——夜结算**就是**这个回报；存进箱里的 180g 是最具体的明日之诺 | 回报即拿即走，过夜失去经济意义，日结算屏退化为流水账 |
| 放下点强度 | 收入揭晓发生在日结算屏 = 设计出来的放下点，回报浓度最高（13-time 同此倾向） | 结算屏只剩复读，放下点变弱 |
| 2 分钟会话推进 | 收获 XP 即时入账（14-progression），入箱金额可见地累积；1 游戏日仅 3 真实分钟，结算永远 ≤3 分钟可达，还可上床提前入睡（13-time 手动睡觉） | 即时反馈更强（唯一优势） |
| 早期流动性 | 收获日上午现金少、补种受限 → 形成「结算前夜的穷一天 + 次日清晨的发薪采购」节拍（见 §6 曲线，D3 穷 / D4 采购），是节奏不是缺陷 | 复利更快（曲线快 ~25%），但抹平了节拍 |
| 实现 | `shippingBin[]` 已在 19-save 存档 schema 中；13-time NightUpdate #1 已预留 `settleShipping` 钩子 | 需推翻两稿假设 |

配套的零焦虑保障（缺一不可）：

1. **白天随时可取回**箱内物品（13-time 已约定），入睡才锁定结算；
2. 箱内物品**永不变质、无容量上限**；
3. 结算自动发生、无需到场点收（睡觉即结算，刷新页面也不丢，见 19-save §结算原子性）；
4. 出售没有「卖错时机」：价格恒定，今天卖和明天卖一个价。

### 2.2 出售通道一览

| 通道 | 位置 | 接受物 | 结算 | 用途 |
|---|---|---|---|---|
| **出货箱（shipping bin）** | 农舍旁（10-world-map `buildings` 层「出货箱底座」） | 作物（M3 起 + 加工品） | 当夜 22:00 | 主通道：收获后顺手一键入箱 |
| **杂货摊 · 卖出页** | 集市角杂货摊（10-world-map (48..51, 19..21)） | 作物 → **寄售**（与出货箱同一份 `shippingBin`，同夜结算）；种子 → **即时退货** | 作物当夜 / 种子即时 | 采购时顺手清背包；买错种子的无损撤销 |

- 「寄售」= 在商店卖出页对作物执行 Shift+左键等操作时，物品进入 `shippingBin` 并显示「今晚结算 ✓」浮字——**不是**即时金币（16-inventory-ui §6.6 需按此修订；其「回购栏」对作物由「白天可从箱中取回」替代，对种子由全额退货替代）；
- 种子退货 = **100% 原价、即时入账**（`refund`）。这是「退货」不是「交易」：买卖零价差 → 无套利、无学习成本、误买零损失（属性测试见 §11-V9）；
- 工具、任务道具不可出售（16-inventory-ui 类别表「不可出售」+ 卖出页双重拦截）。

### 2.3 出货箱交互与 UI（16×16 实体，E 键交互）

```
   ┌─────────────── 出货箱 · 今晚结算 ───────────────┐
   │  箱内（点击取回 ↩）            背包（点击入箱 →）│
   │  ┌────┐┌────┐                ┌────┐┌────┐┌────┐ │
   │  │萝卜││芜菁│                │萝卜││土豆││种子│ │
   │  │ x10││ x2 │                │ x3 ││ x2 ││(灰)│ │
   │  └────┘└────┘                └────┘└────┘└────┘ │
   │  预计入账 252g                [Shift+左键=整堆]  │
   │  ─────────────────────────────────────────────  │
   │  [F] 全部入箱        [Esc] 关闭（界面开启即暂停）│
```

- 「预计入账」实时按 §5.5 计价管线计算（含职业乘数，M3 起含品质）；
- **[F] 全部入箱** = 背包内全部可寄售物一键入箱：「收获 → E 开箱 → F → Esc」四键完成一次出售，满足 2 分钟会话的交互预算（§11-V10）；
- 不可售物（种子/工具）在右侧画暗色不可点（与 16-inventory-ui §6.6 一致）；
- 界面归 16-inventory-ui 的 UI 栈（打开即暂停 tick）。

### 2.4 结算（夜管线 NightUpdate #1，由 13-time 调用、economy 实现）

```ts
export interface SettlementLine {
  itemId: string;
  count: number;
  unitGold: number;          // 经 §5.5 计价管线（floor 后）
  lineGold: number;          // unitGold × count
  firstSale: boolean;        // 该作物首次售出（点亮 collectionLog，M1 即记录、M3 出图鉴 UI）
}
export interface SettlementResult { lines: SettlementLine[]; totalGold: number }

export function settleShipping(state: EconomyState, priceCtx: PriceCtx): SettlementResult;
```

- 原子操作：计价 → `credit` 入账 → 清空 `shippingBin` → 写 `collectionLog` 首售标记 → 发 `economy:sold {itemId, count, gold}`（逐行，供 14-progression 计数器/成就消费）→ 整体先于 autosave 落盘（13-time §结算顺序：`DaySummary.goldBalance` === 存档金币数）；
- 空箱结算 = 零产出，结算屏显示「今天没有出售」（不是错误态，13-time E7）；
- 日结算屏经济区块（屏幕归 13-time，数据由本系统提供 `SettlementResult` + 当日 `GoldChange[]` 聚合）：


## 3. 商店设计：无人杂货摊

### 3.1 形态与原则

- **载体**：集市角的「无人杂货摊」（自助摊位 + 诚实箱，M1 无 NPC 的世界观自洽解；M4 起村民可在摊位附近活动但不参与交易）；交互锚点与 `kind:'shop'` 事件由 10-world-map 提供；
- **营业时间**：全天不打烊（13-time 已对齐）。打开商店即暂停时间——逛街永远不花游戏时间，没有「赶在关门前」的压力；
- **库存**：所有上架条目**无限库存**；
- **反 FOMO 负面清单**（永久承诺，违反即违宪）：无限时折扣、无每日轮换、无随机货架、无会员/礼包、无「仅此一件」。商店目录是 `f(effectiveLevel, purchases)` 的**纯函数**——与真实时间、随机数无关（§11-V6 属性测试）。

### 3.3 解锁节奏与展示

- 每个农场等级在商店至少上新一行（种子或工具）——升级横幅指向商店、商店回应升级，构成「升级 → 上新 → 新目标」的节拍；
- 展示规则沿用 14-progression §2.2：已解锁正常显示（解锁后首个游戏日带 `NEW` 角标）；下一级/下两级条目剪影 + 「Lv N 解锁」（明日之诺）；更远的折叠为「更高等级还有 N 项待解锁」；
- 工具升级行已购显示「已是最高档」；`requires` 前置未满足的（金档先于铜档）显示「需先购买铜档」。

### 3.4 购买事务（界面框架归 16-inventory-ui §6.6，本稿定义规则）

- 交互沿用 16-inventory-ui：左键买 1、右键买 5；实际成交数 `granted = min(请求数, floor(gold / price), 背包可容纳数)`；`granted == 0` 时 toast 给出唯一原因（金币不足 / 背包已满）；
- 首日引导：不赠送种子、不强制留零钱——初始 100g 就是首日种子预算（10 粒小萝卜花光是合法且最优的开局；D3 晨成就 `first_harvest` 的 +20g 保证至少能补种 2 格，见 §6）；
- 工具升级：点击即扣款、即时生效（mechanics §9.4，无「送修等待」）；
- 购买为同步事务（sim 层单入口），UI 层 150ms 去抖防双击重复成交。

```
购买状态机（每次点击）：
 IDLE ─ click(entry, n) ─▶ 校验链
   ├─ 条目剪影（未解锁）──────▶ 不可点（无事件）
   ├─ oneTime 已购 ───────────▶ 「已是最高档」提示
   ├─ requires 未满足 ────────▶ 「需先购买铜档」提示
   ├─ granted = min(n, 可负担, 可容纳)
   ├─ granted == 0 ───────────▶ toast 原因 ─▶ IDLE
   └─ granted ≥ 1 ────────────▶ debit + 发放 + 音效/飘字 ─▶ IDLE
```

### 3.5 ShopApi（提供给 16-inventory-ui 的事务接口）

```ts
// packages/game/src/sim/economy/shop.ts
export type EntryAvailability = 'hidden' | 'locked' | 'available' | 'owned';

export interface ShopEntryView {
  entryId: string; kind: ShopEntry['kind']; price: number;
  availability: EntryAvailability;
  isNew: boolean;                  // 解锁后首个游戏日

}

export interface ShopApi {
  catalog(state: GameState): ShopEntryView[];
  buy(state: GameState, entryId: string, requested: number):
    | { granted: number; cost: number }
    | { blocked: 'locked' | 'owned' | 'requires' | 'no_gold' | 'no_space' };
```

## 4. 定价平衡方法论（与 11-farming 作物表互洽）

### 4.1 基准与公式（宪法 §4.4 定点）

```
基准：T1 = 4.0 g/格·日，每升一档 ×1.5~1.7（T2 ≈ 6.5~7.5，T3 ≈ 9.8~11）

单次作物：  gpd = (sellPrice − seedPrice) / growthDays
再生作物：  gpd_ref = (sellPrice − seedPrice / H_ref) / regrowDays      // 参考口径 H_ref = 4 茬
反推定价：  sellPrice ≈ seedPrice + gpd_target × growthDays
```

> `H_ref = 4` 是**保守校验口径**（宪法 T2≈7.5 / T3≈11 即按此口径得出）：再生作物季初满种的实际收益高于带宽是**设计意图**（奖励长期承诺，mechanics §3.2 法则 3），由不变量 I3 封顶防越档。

### 4.2 新作物准入清单（每个 CropDef 入表前必须全过，CI 固化为 `crop-pricing.test.ts`）

| # | 不变量 | 含义 |
|---|---|---|
| I1 | 单次作物 `sellPrice ≥ ceil(1.5 × seedPrice)` | 单轮毛利率下限，保证「种了就不亏」可感知 |
| I2 | 再生作物 `2 × sellPrice ≥ seedPrice` | 最迟 2 茬回本，季中补种不成陷阱 |
| I3 | 再生作物季初满种 `gpd_max ≤ 1.3 × 档位带上限` | 防再生作物越档碾压下一档单次作物 |

## 5. 计价管线与职业

### 5.1 计价管线（唯一售价计算入口）

```ts
// packages/game/src/sim/economy/pricing.ts
export interface PriceCtx {
  profession: 'horticulturist' | 'artisan' | null;   // M1 恒 null（职业 M3 实装）
}

export function unitSalePrice(item: ItemDef, quality: Quality, ctx: PriceCtx): number {
  let p = item.sellPrice;                  // 基础售价（CropDef 权威字段）
  p *= QUALITY_MULT[quality];              // M1 恒 1；M3：银 1.25 / 金 1.5（宪法 §4.4）
  if (ctx.profession === 'horticulturist' && item.category === 'crop') p *= 1.10;
  if (ctx.profession === 'artisan' && item.category === 'artisan_good') p *= 1.25;  // M3
  return Math.floor(p);                    // 乘完一次 floor，全程整数
}
```

- **先乘后舍只 floor 一次**：金品质 + 园艺师的芜菁 = `floor(36 × 1.5 × 1.1) = 59`，不允许逐乘逐舍；
- 园艺师 ×1.10 与工匠 ×1.25 的数值确认（14-progression §2.4 待确认项）：园艺师对**全部作物**生效、工匠仅对加工品生效，M3 平衡目标是 Lv7（加工棚解锁）后两者期望收益相当——具体校验随 M3 加工品定价稿做；
- **品质不改 XP**（14-progression 已定，防 M3 XP 通胀）；
- 定价校验（§4）一律用基础售价：职业/品质是全局乘数，不参与档位判定。

### 5.2 种子退货价

`refundPrice(seed) = seedPrice`（100%）。园艺师乘数不适用于种子（category 不是 crop），无套利缝隙。

## 6. 收入曲线：基准脚本与逐日推演

### 6.1 基准农夫脚本 R（可直接实现为 sim 快进测试）

```
每个游戏日 6:00 依次执行（全部 sim 层调用，无渲染）：
1. 浇水全部在田作物；
2. 收获全部成熟作物 → 全部投入出货箱；
3. 补种全部空耕地：best = 已解锁 gpd 最高的种子；
   k = max k 满足 price(best)×k + 10×(空格数−k) ≤ gold     // 其余格种小萝卜
   买入并种下（买得起几格种几格，种不满就留空）；
4. 扩张：若耕地 < 等级上限（Lv1=12 / Lv3=18 / Lv5=24，14-progression §2.1），
   锄开新格并用剩余金币种小萝卜；
5. 不买工具（脚本 R 是金流参考线；工具购买见脚本 B）；
6. 入睡 → 夜结算。
```

## 7. 金钱回收 sink 台账

### 7.1 M1（faucet 与 sink 必须闭环）

| 方向 | 项 | 量级 | 时机 |
|---|---|---|---|
| faucet | 作物出售（唯一主水源） | 占总收入 ≥95% | 每夜结算 |
| faucet | 成就一次性金币（14-progression §4.3：#3 +20、#4 +20、#8 +200、#13 +100） | 合计 **340g** | 一次性 |
| faucet | M4 任务奖励（18-ai-quests §8） | ≤24 g/日（默认开，可整体关闭） | M4 起 |
| sink | 种子（循环 sink） | 毛收入的 **35%~45%**（脚本 R：D14 累计支出 1,270 / 毛收 1,368 + 期初 100） | 每日 |
| sink | 工具四件（500×2 + 2,500×2） | **6,000g** 一次性 | D11 ~ 第二月 |

设计结论：M1 一次性 sink 总额 6,000g ≈ 基准玩家 1.5 个月毛收入的 70%——**M1 毕业时金档工具刚好买齐或差一件**，第二月仍有去处；循环 sink（种子）保证现金永远「紧但不卡」。

### 7.2 M3 展望（数值冻结，归 15-building 详设）

| sink | 金额 | 等效总价（材料按 5g 折金） |
|---|---|---|
| 背包 12→24 格 | 1,000g | 1,000g |
| 鸡舍（Lv6） | 2,000g + 木×150 | 2,750g |
| 加工棚（Lv7） | 6,000g + 材料 | ≈7,000g |
| 温室（Lv9） | 15,000g + 木×300 石×300 | 18,000g |
| 农舍升级（纯威望 QoL） | 4,000 → 20,000g | 终局 sink |
| 栅栏/石径/装饰 | 纯材料或小额 | 软 sink（拆除全额返还，零焦虑） |

M3 大额 sink 合计 >50,000g 等效——对应 Lv6+ 产能（32~42 格 + 洒水器）日净 350~600g 约 3 个月的长线，与 14-progression「Lv10 ≈ Day100~115」互恰。

### 7.3 永不出现的回收机制（负面清单）

税收、维护费、作物枯萎损失、工具耐久、能量/体力购买、加速付费、离线惩罚——任何「不玩就变穷」或「玩得慢就变穷」的机制违反零焦虑承诺，列为违宪项。

## 13. 开放问题

1. **Q1（→11-farming）**：作物是否可被玩家主动铲除？若可，E7 软锁证明的「在田作物不会死」前提需补：铲除时返还 50% 种子价或保留救济包为必要防线；
2. **Q2（→11-farming）**：`expectedYield` 恒为 1 的假设需在作物表定稿时确认；任何多收概率都要重算 gpd；
3. **Q3（→13-time/11-farming）**：雨天「假日惊喜」是否在结算屏给一行氛围文案（无数值效果）——纯文案，倾向做；
4. **Q4（→14-progression）**：豆藤解锁在 Lv3 与耕地扩张同级，新增 6 格 + 新作物同日发生，引导上是否过载（本稿曲线显示 D11 支出 210g 可承受，倾向保留）；
5. **Q5（→14-progression / 18-ai-quests，冲突待裁）**：14 稿给 quest XP 的 schema clamp 为 `[5,50]`，但 18 稿 decision 任务 XP=60——二者需对齐（本稿对金币侧无异议：120g ≤ 日收入 12% 安全；建议 clamp 放宽至 `[5,60]` 或 decision 降为 50）；
6. **Q6（→16-inventory-ui）**：卖出页改寄售后，其「回购栏」机制是否整体移除（被出货箱可取回 + 种子退货覆盖），还是保留给 M3 加工品——倾向移除，少一套状态；
7. **Q7（M3 前置）**：工匠 ×1.25 与园艺师 ×1.10 的等价性校验依赖加工品定价（加工时长/增值率），列入 M3 经济补充稿；
8. **Q8（→20-audio-art）**：结算屏入账音与商店购买音是否复用同一金币音色（18 稿奖励已复用商店音）——倾向复用，强化「金币=同一种快乐」。
